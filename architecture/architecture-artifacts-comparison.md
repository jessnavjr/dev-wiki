# Architecture Artifacts Comparison: Enterprise vs Solution vs Systems

This guide compares the typical artifacts generated by enterprise architecture, solution architecture, and systems architecture.

## Quick Definitions

- Enterprise Architecture (EA): Aligns technology direction to business strategy across the organization.
- Solution Architecture (SA): Designs a specific solution that delivers a business initiative.
- Systems Architecture (SysA): Defines the internal technical structure and behavior of a system.

## At-a-Glance Comparison

| Dimension | Enterprise Architecture | Solution Architecture | Systems Architecture |
|---|---|---|---|
| Primary scope | Organization and portfolio | Program, product, or initiative | One system or platform component |
| Time horizon | 2-5 years | 6-24 months | Current and near-term releases |
| Main question | Are we investing in the right direction? | How should this initiative be built? | How should this system be engineered? |
| Typical audience | Executives, portfolio leaders, domain leaders | Product, engineering, security, delivery teams | Engineering teams, SRE, security, operations |
| Level of detail | Strategic and cross-domain | End-to-end solution design | Deep technical and implementation detail |

## Enterprise Architecture Artifacts

These artifacts create strategic clarity and cross-team alignment.

### Common EA artifacts

1. Capability Map
- Shows core business capabilities and maturity.
- Used to identify strategic gaps and platform priorities.

2. Target State Architecture
- Describes desired future architecture across domains.
- Used to align long-term investment and modernization.

3. Transition Roadmap
- Phased path from current state to target state.
- Used to coordinate sequencing, dependencies, and funding.

4. Reference Architecture
- Reusable architecture patterns and standards.
- Used to improve consistency across teams.

5. Technology Standards Catalog
- Approved, tolerated, and retired technologies.
- Used to reduce fragmentation and operational risk.

6. Principles and Guardrails
- Durable decision rules (for example, API-first, secure-by-default).
- Used to enable decentralized decisions with consistency.

7. Portfolio Risk and Technical Debt View
- Enterprise-level risk register and debt hotspots.
- Used for prioritization and governance planning.

## Solution Architecture Artifacts

These artifacts ensure a specific initiative is feasible, coherent, and deliverable.

### Common SA artifacts

1. Solution Context Diagram
- Shows external systems, users, and boundaries.
- Used to clarify scope and integration points.

2. Logical Solution Design
- Major components and interactions.
- Used to align teams on end-to-end design.

3. Integration Design
- API/event flows, contracts, and error handling.
- Used to reduce coupling and integration failures.

4. Data Flow and Data Ownership Model
- Movement of data, transformations, and source-of-truth ownership.
- Used to prevent inconsistency and reconciliation issues.

5. Security and Compliance Architecture
- Authentication, authorization, encryption, auditing, and controls.
- Used to ensure regulatory and security requirements are met.

6. Non-Functional Requirements Specification
- Availability, performance, scalability, resiliency, observability, and cost targets.
- Used to make quality attributes testable.

7. Deployment and Environment Strategy
- Environment topology, release strategy, rollback approach.
- Used to reduce release risk.

8. Architecture Decision Records (ADRs)
- Key decisions with alternatives and tradeoffs.
- Used to preserve rationale and speed future decisions.

## Systems Architecture Artifacts

These artifacts guide engineering of a specific system at implementation depth.

### Common SysA artifacts

1. Component Architecture Diagram
- Internal modules, boundaries, and interfaces.
- Used to structure code and ownership.

2. Runtime and Sequence Designs
- Request paths, asynchronous workflows, state changes.
- Used to validate behavior and failure handling.

3. Interface Specifications
- API schemas, message contracts, versioning strategy.
- Used for compatibility and safe evolution.

4. Data Model and Storage Design
- Entity models, indexing, partitioning, retention.
- Used for performance and correctness.

5. Capacity and Performance Model
- Throughput, latency budgets, scaling limits.
- Used for sizing and reliability planning.

6. Failure Mode and Resiliency Design
- Timeouts, retries, circuit breaking, fallback patterns.
- Used to improve system robustness.

7. Observability Design
- Logs, metrics, traces, alert thresholds, runbooks.
- Used for faster detection and recovery.

8. Operational Readiness Checklist
- Backup/restore, DR posture, SLOs, on-call ownership.
- Used before production launch.

## Artifact Mapping by Architecture Layer

| Layer | Typical artifact outputs |
|---|---|
| Enterprise | Capability maps, target state maps, transition roadmaps, reference architectures, technology standards |
| Solution | Context diagrams, solution designs, integration specs, NFR specs, security architecture, ADRs |
| Systems | Component designs, sequence flows, interface specs, data models, resiliency and observability designs |

## How They Work Together

1. EA sets direction and constraints.
2. SA translates direction into initiative-level design choices.
3. SysA turns solution choices into implementable system designs.

If one layer is missing:
- Missing EA: teams optimize locally and drift strategically.
- Missing SA: initiatives become inconsistent and hard to deliver.
- Missing SysA: builds are fragile, costly to operate, and hard to scale.

## Practical Guidance for Tech Leads

Use this quick heuristic:

- Ask EA questions when deciding long-term platform direction, standards, or cross-domain investment.
- Ask SA questions when shaping an initiative with multiple systems and stakeholders.
- Ask SysA questions when refining internals, performance, reliability, and operability.

For one major project, maintain at least:

1. One EA alignment note (standards, principles, target-state fit).
2. One SA packet (context, integration, data, NFRs, security).
3. One SysA packet (components, interfaces, resiliency, observability).

That minimum set prevents strategy drift, design ambiguity, and implementation risk.
